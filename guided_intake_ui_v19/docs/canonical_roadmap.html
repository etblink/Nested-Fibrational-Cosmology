Governing Law

The canonical completeness arc

Iterations 190–212 distil a precise sequence of properties a suite must have before it is canonical in the strongest sense — not merely complete in content, but complete as a transferable, reproducible, governed mathematical object. Each iteration proves the weakest next theorem, isolating exactly what turns the prior property into the next one.

Every phase of this roadmap is anchored to a theorem in the arc. No phase is "done" until the corresponding property is satisfied, not merely approximated.

Iter. 204
Canonical Instantiation
The minimal core {A, D, I*, J*, K*} is sufficient to lawfully regenerate the full governed suite without hidden judgment.
Iter. 205
Canonical Handoff
A new competent steward can reconstruct and govern the suite from the core without relying on the original author's memory, history, or unwritten interpretation.
Iter. 206
Canonical Replication
Independent competent stewards given the same core reconstruct the same governed suite state — same status map, entry order, admissibility law, and outward compression.
Iter. 207
Presentation-Invariance
Any admissible re-expression (dossier, cover note, abstract, synopsis) preserves the same status map, dependency order, admissibility boundary, and compression layer.
Iter. 208
Canonical Delta
Two successive authoritative suite states differ only by the explicit ledger-certified revision event; no change is introduced by packaging, summary, or steward choice.
Iter. 209
Canonical Replay
Every authoritative state transition is reconstructible step-by-step from the ledger-certified transition record alone — evolution is not just auditable but reproducible.
Iter. 210
Canonical Execution
The current authoritative state together with the governed action taxonomy determines which kinds of next moves are admissible, without hidden author-dependent judgment.
Iter. 211
Canonical Verification
Every admissible proposed move has a named, move-specific pre-authoritative check condition determinable from the canonical record before it can change the suite state.
Iter. 212
Canonical Authorization
A verified move changes the suite only by an explicit ledger-governed authorization act. Verification is necessary but not sufficient; passing the check does not itself update the canon.
Standing governance law (all phases). Paper D forbids scope enlargement without an explicit transfer theorem. Paper I makes the ledger the sole source of truth for claim status. Papers K and L are non-enlarging packaging layers, not new theorem sources. No presentation, summary, or outward communication may function as a hidden authorization channel.

Phase II

Canonical handoff preparation

Iter. 205 — Canonical Handoff
Target theorem

A new competent steward, given only the canonical core, can reconstruct and govern the suite without relying on the original author's memory, prior conversation history, or unwritten interpretation.

This phase's test is the two-steward experiment: hand two independent competent mathematicians only K*, J*, A, D, I8, and BR. Ask each to determine: (1) what is theorem anchor, (2) what is open bridge, (3) what is computational vs. unconditional, (4) what is admissible outward compression, (5) what currently governs status. If both reach the same answers without further input, handoff is satisfied.

Handoff requirements (from Iter. 205)

  • Author-independence: All status-governing, admissibility-governing, and routing-governing facts are encoded in the core and its lawful reconstructions — nothing in unwritten author memory.
  • Conversation-independence: No prior chat history, oral explanation, or bespoke interpretation is needed to answer the five canonical questions above.
  • Reconstruction determinacy: Two competent stewards given the same core reconstruct the same governed suite state up to the already-licensed succession and ledger rules.
  • Governed continuation: The new steward can not only read the canon, but also perform lawful future actions — upgrade, downgrade, reclassify, package, transmit — under the same doctrine and ledger discipline.

Concrete assembly tasks

  • Ensure K* contains an explicit, self-contained statement of the intake law — the first-pass reading path for serious readers — that does not presuppose familiarity with any prior version.
  • Ensure I8 contains a self-standing revision protocol section: what events change ledger status, what the ledger entries look like, and what makes a public statement provisional vs. authoritative.
  • Ensure D2 contains a self-standing evidentiary law statement: what certificates may and may not do, what makes computation theorem-usable, and what the anti-smuggling boundary is.
  • Verify J2 does not rely on implicit knowledge of prior brief generations (J or J_early) to be correctly read. If J2 references a "previous brief," check whether that reference is still live or has been superseded.
  • Write a one-page handoff note (candidate for inclusion in K*) summarising: live slot occupants, ledger generation, branch status, and the five canonical questions. This note is the functional test object for Iter. 205.
Phase III

Canonical replication and presentation-invariance

Iters. 206–207
Target theorems

Replication (206): Independent competent stewards given the same core reconstruct the same governed suite state — same status map, entry order, admissibility law, compression layer. Presentation-invariance (207): Any admissible re-expression preserves that governed state exactly.

Replication means the five replication properties below are jointly satisfied by the core. Presentation-invariance means that L* outward forms, K* dossiers, and any future branch synopses cannot introduce governed differences through paraphrase or reordering.

Five replication requirements (Iter. 206)

  • Status replication: I8 is the unique status source of truth; later public language defers to it. No paper may carry its own independent claim-status statement that could diverge.
  • Entry replication: K* fixes the minimal serious package and the canonical first-pass routing law. No alternative entry path is legitimate unless K* names it.
  • Admissibility replication: D2 fixes the evidentiary boundary. No steward discretion may extend what computation can prove.
  • Compression replication: K2 and L2 are non-enlarging packaging layers. They summarise and transmit; they do not create new governed content.
  • No hidden steward freedom: Any difference between two reconstructed suite states is traceable either to a ledger-certified succession event or to steward error — not to legitimate interpretation plurality.

Presentation-invariance checklist (Iter. 207)

  • Status invariance: Audit L2 (outreach kit) and any cover letters for paraphrase that strengthens conditional, computational, prototype, or open-bridge claims. Flag any instance where language implicitly upgrades status.
  • Dependency invariance: Confirm all public-facing reading guides route through A→D→I8→J*→K* before frontier papers. No external synopsis may suggest reading branch papers before the core.
  • Admissibility invariance: Check that no summary or abstract implies bounded computational success constitutes asymptotic, continuum, or physical-interpretive closure.
  • Compression invariance: Confirm L2 is explicitly governed by and subordinate to A, D, I8, J*, K*. Any abstract or letter derived from L2 should cite the live ledger generation.
  • Flag paper_NFC_CLU, paper_NFC_EFE, paper_NFC_LS2, paper_NFC_3Gen, paper_NFC_Weyl, paper_NFC_Stage0 — each must be visibly indexed as a branch discharge note, not an alternative Start Here paper, to preserve entry replication.
Phase IV

Canonical delta visibility and replayability

Iters. 208–209
Target theorems

Delta (208): Successive authoritative suite states differ only by the explicit ledger-certified revision event; no change is introduced by packaging or steward choice. Replay (209): Every authoritative state transition is reconstructible step-by-step from the certified transition record, using the fixed serious base {A, D, I*, J*}.

Ledger transition record

The ledger update chain I2→I3→I4→I5→I6→I7→I8 represents eight successive authoritative states. For each transition, the delta must be explicitly stated in the successor document. The replay test: given In and the In+1 transition record, a competent steward can reconstruct In+1 without additional inputs.

  • Verify each ledger generation (I2 through I8) carries an explicit delta section: which claims changed status, which bridge was discharged, which promotion event occurred.
  • Confirm that I8 (terminal) either (a) carries a final-state certification or (b) explicitly names the remaining open bridge obligations without suggesting closure.
  • Check that no packaging difference between ledger generations introduces apparent claim changes that are not ledger-certified (e.g., removal of a claim from I7 to I8 that was not explicitly reclassified or retired).
  • Audit the paper_O2_substrate_independence and paper_D2_nfc_certification additions — confirm their contribution to the governed suite is captured in a ledger delta entry in I8 or later.
  • Ensure the ledger's revision protocol section (required from Phase II) is sufficient to replay any transition from an archived state. If it is not, extend it.
Phase V

Canonical execution and verification

Iters. 210–211
Target theorems

Execution (210): The current authoritative state and the governed action taxonomy together determine which next move classes are admissible, without hidden judgment. Verification (211): Every admissible proposed move has a named pre-authoritative check condition determinable from the canonical record before it changes the suite state.

Governed action taxonomy (from Iter. 210)

Every serious next move belongs to one of the following named classes. This taxonomy should be explicitly published — ideally in K* or a dedicated governance annex:

Action class Description Verification condition (Iter. 211)
theorem-upgrade A new unconditional proof advances a currently conditional or open claim Proof satisfies suite dependency and status law; no scope enlargement without transfer theorem
bridge-discharge A new theorem closes a named open bridge entry in the ledger Theorem explicitly closes the named bridge entry; ledger entry can be retired
D-compliant promotion A soundly scoped certificate plus matching transfer theorem enters theorem flow Certificate is soundly scoped; transfer theorem already proved; no tacit scope enlargement
reclassification An existing entry is refactored, downgraded, or its status is revised Explicit ledger revision justified by current theorem stack; no claim upgraded by paraphrase
intake/transmission refresh K* or L* is updated to reflect the current ledger state Non-enlarging packaging update; subordinate to A, D, I*, J*, K*; no new claim introduced
no-op / stable non-upgrade A proposed action that does not pass its verification condition Governed refusal; suite state is unchanged; no ambiguous quasi-state
  • Publish the action taxonomy above (or an equivalent statement) in K* or a governance annex, making it accessible without reading the full iteration dialogue.
  • Ensure I8's revision protocol maps each ledger entry type to one of the action classes above.
  • Confirm D2 explicitly states the verification condition for D-compliant promotion: soundly scoped certificate + already-proved transfer theorem. The condition should be checkable without author input.
  • Review paper_NFC_CLU, paper_NFC_EFE, and paper_NFC_LS2 — determine which action class each represents and confirm each has a corresponding I8 delta entry. If not, they are in pre-authorization limbo and must be marked provisional.
Phase VI

Canonical authorization

Iter. 212 — Canonical Authorization
Target theorem

A canonically verified move changes the suite only when its successful verification is followed by an explicit ledger-governed authorization act — and not before. Verification is necessary but not sufficient.

This is the terminal governance theorem in the arc. It separates the mathematical event (a proof is found, a certificate is computed) from the governed state change (the ledger is updated). Papers may circulate, be submitted, even be accepted by referees, without altering the authoritative suite state. The suite state changes only by ledger entry.

Authorization requirements

  • Verification ≠ authorization: Passing the move-specific check condition does not update suite status. A checked but not yet ledger-entered result is provisional only.
  • Explicit authorization events: The suite recognizes four authorization acts — proof-upgrade entry, bridge-discharge entry, D-compliant certificate promotion entry, and explicit reclassification/factorization entry. No other event changes the canon.
  • Authorization-source determinacy: Whether authorization has occurred is decidable from the canonical record alone — not from circulation, submission status, author intent, or social uptake.
  • Pre-authorization limbo policy: All results currently in circulation that do not yet have corresponding I8 (or later) ledger entries are provisionally discussed but not yet suite-governing. The suite presentation must clearly communicate this distinction.
  • Audit publicly circulating NFC papers (paper_NFC_3Gen, paper_NFC_Weyl, paper_NFC_Stage0, paper_NFC_CLU, paper_NFC_EFE) for ledger authorization status. Any paper with a result not yet in I8 must be displayed as pending authorization, not as a settled advance.
Canonical completeness criterion. The suite reaches canonical completeness when all six phases are satisfied: live slot assignments are verified (I), handoff conditions hold (II), replication and presentation-invariance hold (III), delta and replay are fully documented (IV), the action taxonomy and verification conditions are published (V), and the authorization boundary is explicit and enforced (VI). At that point the governed program becomes not just mathematically substantive but fully transferable, reproducible, and institutionally durable.
