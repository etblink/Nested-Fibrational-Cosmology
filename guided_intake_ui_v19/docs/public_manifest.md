# Dream Paper Suite — Public Manifest

Date: March 21, 2026
Project Lead: Evan Thomas Kotler

## Purpose
This manifest defines the public-facing package order for the current Dream Paper / NFC release stack. It is meant to prevent mis-entry into the suite and to preserve theorem anchor, claim status, and dependency order.

## Default reading order
1. **`dream_suite_intake_packet_v1.pdf`**  
   External intake guide. Read first.
2. **`dream_paper_kernel_public_v1.pdf`**  
   Public mathematical anchor. Self-contained finite kernel paper.
3. **`dream_paper_v25.pdf`**  
   Program synthesis paper. Architecture, burden-routing, and branch map.
4. **`dream_suite_v5_final.html`**  
   Portal / navigation layer.
5. Historical materials only after the above are understood.

## Artifact roles
### 1. Intake packet
- File: `dream_suite_intake_packet_v1.pdf`
- Role: governed external entry path
- Function: tells serious readers how to enter the suite without silently upgrading claims

### 2. Kernel-only public paper
- File: `dream_paper_kernel_public_v1.pdf`
- Role: public theorem anchor
- Function: isolates the finite theorem-bearing core from bridge-heavy downstream claims

### 3. Program synthesis paper
- File: `dream_paper_v25.pdf`
- Role: architecture and dependency document
- Function: explains how the theorem-bearing core, universal/source architecture, and branch programs fit together
- Warning: not a standalone closure proof of all downstream physical conclusions

### 4. Portal
- File: `dream_suite_v5_final.html`
- Role: public navigation layer
- Function: presents the package as a kernel-first research program rather than a terminal closure document

## Status law
All claims should be read under exactly one of these force classes:
- **Unconditional** — proved within the finite kernel package
- **Conditional on named inputs** — depends on explicit bridge theorems / hypotheses
- **Computational / certificate-supported** — supported at stated scope by finite computation or verified certificate transfer

No summary, portal page, or pitch document should silently upgrade a claim across those classes.

## Dependency order
The package should be read in this order:

kernel theorem anchor -> source architecture -> named bridge inputs -> branch programs

Yang–Mills, Navier–Stokes, structural Einstein, and wider physical interpretations are downstream branch programs, not first-contact proof objects.

## Public standing
The strongest settled public mathematical object in this package is the kernel-only public paper. The synthesis paper is public-facing only as a status-controlled architecture document.

## Archive policy
Older terminal-shell materials are retained only for historical continuity and internal research context. They are not the default public face of the suite.
